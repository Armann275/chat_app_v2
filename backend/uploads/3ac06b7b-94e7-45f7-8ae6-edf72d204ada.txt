Integration Overview
The purpose of this section is to provide basic description of Amusnet Multi Game Server (MGS) integration with different platform.

Glossary¶
   MGS – Amusnet Interactive Multi Game Server
   Casino Operator – Amusnet Interactive MGS Client - Casino Operator
   API – Application Program Interface
   Optional parameter – Parameter which is required only for specific jurisdictions. It can be added to standard protocol depending on the market where Casino Operator operates
   Bonus – The Free Spin feature
   Bundle – The awarded bonuses by one API request to one Player
   Free Game Server – Instance of MGS available for free play
   FGLP – Free Game Launch Protocol

Integration Process¶
Creating a testing environment¶
The endpoints provided by Casino Operator should be implemented according to the following example:


https://{casino-operator-domain}/v1/auth

https://{casino-operator-domain}/v1/withdraw

https://{casino-operator-domain}/v1/deposit

https://{casino-operator-domain}/v1/withdraw-deposit

https://{casino-operator-domain}/v1/award-deposit

https://{casino-operator-domain}/v1/buy-in/withdraw

https://{casino-operator-domain}/v1/buy-in/deposit
Once the endpoints are implemented and provided to the MGS testing process on MGS Staging environment can start from both sides.
Once testing is finished, the communication can go to Production environment.
For Production environment, a new Amusnet MGS domain name will be provided.

Important Note

Amusnet Multi Game Server supports games which allow a single Game Round to contain more than one Withdraw and Deposit request.

Steps¶
Step 1 - The Player clicks on the play link of а game or а Portal integrated in the Casino Operator website
Step 2 - Casino Operator website generates Game Launch link
Step 3 - The Casino Operator generates a DefenceCode
Step 4 - The Casino Operator encloses the DefenceCode to the Game/Portal launch URL (HTTPS request) along with other parameters and sends it towards Amusnet MGS
Step 5 - Casino Operator website generates Game Launch request to Amusnet MGS
Step 6 - An Authentication Request is made by Amusnet MGS towards the Casino Operator using the DefenceCode provided by the HTTPS request parameters
Step 7 - The Casino Operator generates an AuthenticationToken which will be used for multiple Game Authentications inside the Portal
Step 8 - The Casino Operator sends an Authentication Response containing the player's available balance and the AuthenticationToken
Step 9 - Game is shown to the Player
Step 10 - The Player chooses an amount and makes a Bet on a chosen Game
Step 11 - Withdrawal Request is initiated by Amusnet MGS. This method requests the Bet amount from the Player's account with the Casino Operator
Step 12 - The Withdrawal Response should contain CasinoTransferId and updated balance
Step 13 - The outcome for this specific bet is generated. A Deposit Request is initiated by Amusnet MGS. This request transfers the amount won in the Game to the Player's account with the Casino Operator
Step 14 - The Deposit response should contain CasinoTransferId and the updated balance
Step 15 - The result is shown to the Player
Step 16 - The Player clicks on another Game from inside the Portal
Step 17 - Previously generated authentication token will be supplied as cookie in the browser
Step 18 - An Authenticate request is made by Amusnet MGS towards the Casino Operator using the AuthenticationToken provided by the HTTPS request parameters
Step 19 - The Casino Operator validates the AuthenticationToken
Step 20 - The Casino Operator sends an Authenticate response containing the Player's available balance
Step 21 - Game is shown to the Player
Step 22 - The Player chooses an amount and makes a bet on a chosen Game
Step 23 - Withdrawal Request is initiated by Amusnet MGS. This method requests the bet amount from the Player's account with the Casino Operator
Step 24 - The Withdrawal Response should contain CasinoTransferId and updated balance
Step 25 - The outcome for this specific bet is generated. A Deposit Request is initiated by Amusnet MGS. This request transfers the amount won in the Game to the Player's account with the Casino Operator
Step 26 - The Deposit response should contain CasinoTransferId and the updated balance
Step 27 - The result is shown to the Player


Security Model¶
For protection of both the Casino Operator and the Amusnet MGS, the following methods are used: IP filtering and predefined username and password of each request for mutual authentication.

Username and Password¶
To establish Casino Operator system - Amusnet MGS HTTPS authentication the Amusnet MGS predefines a username and password for the Casino Operator.
The Amusnet MGS encloses the username and password in each request to the Casino Operator system. The username and password are presented as Header: Authorization; auth-scheme: Basic; encoding: Base64 in each request.

IP Detection¶
Amusnet MGS provides the Casino Operator with a list of authorized IP addresses. Only HTTPS requests originating from these IP addresses must be handled. Connection attempts from any other IP address must be rejected.

Game Play Integration¶
Game Launch URL¶
When a Player clicks a link on the Casino Operator website to launch a Game, the HTTPS request is generated. The Casino Operator's system transmits data to the Amusnet MGS system within a URL string. The data uniquely identifies the Player, the Currency, the selected Game, the Casino Operator system and the Portal. These and other parameters allow Amusnet MGS system to authorize and authenticate the Player and launch the selected Game.

During this operation, the resulting Game Launch URL should be looking similar to this:


https://{amusnet-mgs-domain}/MGL?defenceCode=25cac65913564a6178526ecf0e95212d&playerId=11111&portalCode=SomeOperatorId_EUR&language=en&country=bg&gameId=801&screenName=1111&hasPromo=true
Mandatory Parameters¶
defenceCode - secure code which is a unique string generated by the Casino Operator system, used for Portal launch and Authenticate; it can be used only once
playerId - unique ID string for specific Player assigned by the Casino Operator system
portalCode - predefined constant that binds Portal and Currency, provided by the Amusnet MGS system
language – language code (ISO 639-1)
country – the country which the Player provided at registration with the Casino Operator (ISO 3166-1 alpha-2)
gameId – unique internal number that refers to a specific game; launch with a specific game
Optional Parameters¶
screenName – the visual representation of the Player's nickname; if none the default value is player
hasPromo – a boolean flag that indicate whether the player is eligible to participate in promotions; if the parameter is not provided, then the player is considered eligible for promotions participation; possible values true or false
Parameter Restrictions¶
Parameter	Type	Maximum Length	Disallowed characters
defenceCode	Text	192 characters	
session_id	Text	128 characters	
screenName	Text	128 characters	
playerId	Text	64 characters	#
authenticationToken	Text	192 characters	
Important Security Note

Every time when a Player attempts to launch a game from Casino Operator site, the Casino Operator must generate a new defenceCode. The defenceCode is used only once for authentication and for generation of AuthenticationToken (which is reusable) and when the authentication is done, the defenceCode becomes invalid and nonreusable. If there is no Authentication request 2 minutes after the defenceCode is generated, the defenceCode must be ivalidated. It is recommended that if there is no Authentication request with this AuthenticationToken 6 hours after it is lastly used, this AuthenticationToken expires.
Amusnet MGS system does not validate the defenceCode reuse by any means!

Mobile Game Launch URL¶
When the Casino Operator detects that the Player is launching games from a mobile device, a client parameter should be added in Game Launch URL:


https://{amusnet-mgs-domain}/MGL?defenceCode=25cac65913564a6178526ecf0e95212d&playerId=11111&portalCode=SomeOperatorId_EUR&screenName=11111&language=en&country=bg&gameId=801&client=mobile&closeurl=someurl
Mandatory Parameters¶
client – specifies the type of application - web or mobile - in order to launch the appropriate game content; the expected values are either client=desktop for desktop content, or client=mobile for mobile version
Optional Parameters¶
closeurl – URL value used for redirection to Casino Operator page, when Player clicks on the Home button within a Game. In case not provided, close tab function is executed. If the game is launched in an iframe you can pass com.egt-bg.postMessage in which case a postMessage will be sent to the parent window.
In case when Game is launched in an iframe, the following JavaScript code should be placed in the page where the iframe referring to Amusnet MGS is embedded:


Implement custom logic here to be executed when the user presses the exit button of the GamePlatform

function onExitGamePlatformEGT() {
    ... your custom exit logic here
}

There is additional button for reloading the game when no connection occurs. You can implement what happens when the user presses the reload button here

function onReloadGamePlatformEGT() {
    .... your custom reload logic here
}

function receiveMessage(event) {
    if (event.data) {
        if (event.data.command == 'com.egt-bg.exit') {
            onExitGamePlatformEGT();
        }
        else if (event.data.command == 'com.egt-bg.reload') {
            onReloadGamePlatformEGT();  
        } 
    }
}

window.addEventListener("message", receiveMessage, false);

Game Display¶
We recommend launching desktop games in an iframe with the following attribute present allow="autoplay; fullscreen";. If the games are not launched this way, they will not be able to go fullscreen and autoplay sounds on some browsers will start muted. If you don't allow fullscreen for the iframe, we will send you the event 'requestFullscreen' when the user wants to enter fullscreen and 'cancelFullscreen' when the user wants to exit fullscreen. Then you can call the appropriate browser APIs. Note that in this case we expect to receive this event


{
    name: "fullscreenChanged",
    isFullscreen: <boolean telling use if the browser is in fullscreen or not>
}
This will allow us to properly update the UI in the game.
When launching an Amusnet MGS Game, the default container's resolution must be 1280x720 (16:9).

Jackpot Banner Component¶
Implementing Amusnet MGS Online Jackpot Banner and requesting current values from the Amusnet (MGS) system requires a one-time development effort by the Casino Operator software providers.

Amusnet MGS is providing a ready-to-use HTML component for the Jackpot Banner which can be found as follows:

Jackpot Cards HTML components
Golden Coins Link HTML components
Egypt Quest HTML components
In order to be able to use this component, the following steps should be executed:

Step 1: Extract all files from zip file
Step 2: Open index.html with preferred text editor
Step 3: In the <body> tag there is already an example of how to use and configure Amusnet MGS Jackpot banner
The <div> id must be defined as id="com-egt-jackpot-html-jackpotBanner"
com-egt-jackpot-styles.css, located in css/com-egt-jackpot-styles.css and com-egt-jackpot-utilities.js located in js/com-egt-jackpot-utilities.js must be included
jquery-3.7.1.min.js must be included
Step 4: Create object with EgtJackpotComponent class with following configuration properties:


jackpot = new EgtJackpotComponent(
    {
        URL: 'https://{amusnet-mgs-domain}/jackpot-stats/statistics/{UID}',lang: 'en', resourceURL: '', redirectURL: '', maxDigits: [6, 7, 8, 9], requestInterval:[5000, 5000, 15000, 30000, 60000, 180000], xtra_small: 300, small: 650, medium: 1050, large: 1400
    }
);
Key / value pairs¶
URL - Jackpot feed URL provided by Amusnet to the Casino Operator
lang - specified the language of the jackpotComponent described in Languages XML
resourceURL - path to the Amusnet Jackpot Banner if it is not located in the default assets folder
redirectURL - URL to redirect to once the banner is clicked
maxDigits - the maximum amount of digits that will be shown on each banner box. e.g. Clubs will show only 6 digits , Diamonds Will show 7 digits etc.
requestInterval - request delay in MS. After 180000 is reached each following request will be sent with that delay
xtra_small - Breakpoint width for extra small devices - e.g. old phones, with maximum width of 240px
small - Breakpoint width for small devices - for modern phones
medium - Breakpoint width for medium devices - for tablets and small laptop screens
large - Breakpoint width for large devices - for laptops and big monitors

Bonus Platform
Amusnet Bonus Platform provides Casino Operator with the option to award bonuses to their players. In order to use Amusnet Bonus Platform, a Casino Operator and Amusnet should perform integration of the additional feature API.

Important Note

Implementing the Bonus Platform does not automatically make this functionality available for Casino Operator. Amusnet MGS must also develop it's part of the integration and set all required configurations.

Bonuses can be played on any bet level and do not influence the game's Return to Player (RTP) percentage.

Any game play made with Free Spins is not included in the total bet and total win made on Amusnet games for Casino Operator.

The optional parameters that serve Amusnet Bonus System integration can be checked in the WithdrawAndDeposit, Withdraw and Deposit requests. They are within the key bonusData and if the player has been awarded with bonus, this tag will be present in the requests until player exhaust the bonus or Promotion has been expired. Please be informed that for players who do not have bonuses, Amusnet Multi Game server will send requests without the above-mentioned optional parameters.

Terms and Conditions¶
Amusnet MGS should enable Bonus Platform functionality per operator (sub-system, skin) to kick-off Bonus Platform integration activities
Multiple Promotions per Game may be created and run at the same time
A Promotion may support more than one currency
Important Note

Once a Promotion reaches its Start Date, it is no longer part of the forthcoming Promotions. When a Promotion with Start Date in the future is terminated, it is not interpreted as forthcoming.

Bonus Platform integration endpoint:


https://staging-pod.games.amusnet.io/bonus-platform/operator/v1
Bonus Platform Prod endpoint:


https://{amusnet-mgs-domain}/bonus-platform/operator/v1
Amusnet reserves the right to choose which games are eligible for bonus.
Amusnet reserves the right to terminate a Promotion if any suspicious behaviour is detected.
